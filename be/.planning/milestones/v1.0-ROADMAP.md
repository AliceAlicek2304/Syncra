# Milestone v1.0: Stability

**Status:** SHIPPED 2026-04-27
**Phases:** 1-3
**Total Plans:** 12

## Overview

The v1.0 Stability milestone focused on securing the billing pipeline, optimizing request-response cycles, moving processing to the database layer, decoupling service logic from environment variables, and ensuring long-term stability through comprehensive testing and observability.

## Phases

**Goal:** Secure the billing pipeline and optimize the request-response cycle.
**Depends on:** None
**Plans:** 4 plans

Plans:

- [x] 01-01: Establish baseline integration tests for Webhooks and Middleware

- [x] 01-03: Refactor Stripe Service to remove global state
- [x] 01-04: Implement Redis caching for Tenant Resolution

## Milestone Summary

**Key Decisions:**

- Use `IdempotencyRecord` for Stripe events with `stripe_event_{id}` key
- `User:Workspace` pair caching for tenants with 1h expiration
- Robust JWT authentication in DI to handle empty secrets in tests
- Mocked infrastructure (Hangfire, DB, Redis) for stable integration tests
- EF Core 8 Queryable translation for `ScheduledTime` value object (explicit cast to `DateTime?`)
- Lightweight `Result<T>` pattern instead of third-party libraries
- `IPlanRepository` to encapsulate Plan querying by Stripe identifiers
- Precedence-based integration health reporting
- Standardized `ValidationException` for domain value object invariants

**Issues Resolved:**

- Fixed context overflow at 100+ phases (not applicable — milestone had 3 phases)
- Resolved 190+ compilation errors in test project
- Fixed SQLite NOT NULL constraint on `ScheduledAt` for `ScheduledTime.None`
- Fixed test data seeding Foreign Key errors in SQLite
- Fixed missing Stripe-Signature header handling (500 → 400)
- Fixed domain value objects throwing `ArgumentException` instead of `ValidationException`

**Technical Debt Incurred:**

- Seed data contains placeholder Stripe IDs (`prod_placeholder_free`, etc.) requiring production update
- Some outdated Application-layer tests temporarily disabled with `#if FALSE` during Result pattern transition
